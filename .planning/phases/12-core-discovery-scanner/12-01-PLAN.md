  <action>
Create `skills/toolkit-discovery.sh` as an executable bash script. Use `set -euo pipefail`. The script must:

**1. TTL Cache Check (PERF-01)**
- Registry path: `.claude/toolkit-registry.json` (project-local)
- Check if registry file exists and its modification time is within 3600 seconds (1 hour)
- Use `stat -f %m` on macOS, `stat -c %Y` on Linux for file mtime (try macOS first, fallback to Linux)
- If cache is fresh AND `--force` flag is NOT set: extract compact summary from cached registry using python3 (read JSON, build summary), print to stdout, exit 0
- If cache is stale, missing, or `--force`: proceed to full scan

**2. Agent Scanning (DISC-01)**
- Scan `~/.claude/agents/*.md`
- If directory does not exist, set agents to empty array
- For each .md file, extract YAML frontmatter (text between first `---` and second `---`)
- Parse frontmatter fields: name, description (may be multi-line with `>`), model, tools (YAML list), maxTurns
- Use python3 for YAML-like parsing: read frontmatter block, extract fields with simple line parsing (do NOT require PyYAML -- use string splitting on `:` for single-line fields, handle `>` continuation for description, parse `- item` lists for tools)
- Build JSON entry: `{ name, type: "agent", description, stages: [], source, model, tools: [], maxTurns }`

**3. Skill Scanning (DISC-03)**
- Scan `~/.claude/skills/*/SKILL.md` (directories containing SKILL.md)
- Also scan `~/.claude/skills/*.md` for standalone skill files (like wizard.md, wizard-backing-agent.md) -- these are NOT skills for the registry; skip standalone .md files that do NOT have a SKILL.md in a subdirectory
- If directory does not exist, set skills to empty array
- For each SKILL.md, extract YAML frontmatter: name, description
- Build JSON entry: `{ name, type: "skill", description, stages: [], source, trigger: name }`

**4. Hook Scanning (DISC-04)**
- Scan `~/.claude/hooks/*` (all files, .sh and .js and others)
- If directory does not exist, set hooks to empty array
- For hooks, derive name from filename (strip extension), and attempt to determine the event type by cross-referencing with `~/.claude/settings.json` hooks section
- Read `~/.claude/settings.json`, extract the hooks object, and for each registered hook find which event it belongs to (SessionStart, PostToolUse, etc.) by matching the hook filename in the command string
- Build JSON entry: `{ name, type: "hook", description: "{event} hook", stages: [], source, event }`

**5. MCP Plugin Scanning (DISC-02)**
- Scan TWO sources for MCP plugins and merge results, deduplicating by name:
- **Source A: `~/.claude/settings.json` enabledPlugins** -- extract the `enabledPlugins` object. Each key is formatted as `{name}@{marketplace}` with boolean value. Only include plugins where value is `true` (enabled).
- **Source B: `~/.claude/installed_plugins.json`** -- if this file exists, read and parse it. Extract plugin entries (the structure may be an array of objects or a map -- inspect the actual file structure and adapt). Extract the plugin name from each entry.
- Merge both sources: combine all discovered MCP plugin names. Deduplicate by name (if a plugin appears in both enabledPlugins and installed_plugins.json, include it once).
- Build JSON entry for each unique MCP plugin: `{ name: (extracted name), type: "mcp", description: "{name} MCP plugin", stages: ["research", "planning", "execution", "review"], source: "enabledPlugins" }` (source: "enabledPlugins" if from settings.json, "installed_plugins.json" if only from marketplace, "enabledPlugins" if from both)
- **CRITICAL per CONTEXT.md locked decision:** ALL MCP entries get `stages: ["research", "planning", "execution", "review"]` -- MCP descriptions are too minimal for reliable keyword matching, so they bypass stage tagging and receive all four stages unconditionally.

**6. Stage Tagging (MATCH-01)**
- After all entries are collected, apply keyword matching to each NON-MCP entry's description field (MCP entries already have all stages assigned in step 5 -- skip them here)
- Four stage keyword lists (pipe-delimited for grep -iE):
  - research: `research|investigate|analyze|explore|scan|audit|inspect|fetch|crawl`
  - planning: `plan|design|architect|strategy|roadmap|scope|requirement|scaffold`
  - execution: `execute|implement|build|write|deploy|create|generate|fix|develop`
  - review: `review|test|validate|verify|check|lint|quality|security`
- For each entry, test description against each stage's pattern. If match, add stage to stages array
- Also match against the name field as a secondary signal (e.g., "code-reviewer" should match review)
- **Zero-match fallback per CONTEXT.md locked decision:** Entries with no keyword match get ALL four stages assigned: `["research", "planning", "execution", "review"]` -- better to over-surface than miss a useful tool

**7. Registry Write (DISC-05)**
- Build complete registry JSON: `{ version: 1, scanned_at: ISO_TIMESTAMP, counts: {agents, skills, hooks, mcp}, agents: [...], skills: [...], hooks: [...], mcp: [...] }`
- Ensure `.claude/` directory exists (`mkdir -p .claude`)
- Write to `.claude/toolkit-registry.json` using python3 json.dumps with indent=2
- Use python3 for ALL JSON construction (not heredoc cat) -- the registry is complex with arrays of objects containing arrays, and proper escaping of description strings with quotes/special chars is critical for 160+ agents

**8. Compact Summary (MATCH-02)**
- After writing registry, build compact summary JSON for stdout
- Summary: `{ version: 1, counts: {agents, skills, hooks, mcp}, by_stage: {research: [...], planning: [...], execution: [...], review: [...]} }`
- Each stage list: collect all entry names that have that stage, then sort with priority: entries whose source contains "keystone" or whose name matches known Keystone agents (wizard, bmad-gsd-orchestrator, context-health-monitor, phase-gate-validator, doc-shard-bridge, project-setup-wizard, project-setup-advisor, it-infra-agent, godot-dev-agent, open-source-agent, admin-docs-agent, stack-update-watcher) first, then GSD-related names, then alphabetical
- Cap each stage list at 8 names (MATCH-02)
- Target total summary size: ~500-600 bytes
- Print summary JSON to stdout (this is what wizard-detect.sh will capture in Phase 13)

**Implementation approach:**
- Use a single python3 inline script (invoked via `python3 -c "..."`) for the heavy lifting: YAML frontmatter parsing, JSON construction, stage tagging, and summary generation. The bash wrapper handles: argument parsing (`--force`), TTL cache check, directory existence checks, and calling the python3 block
- This mirrors the pattern in wizard-detect.sh where python3 is used inline for JSON manipulation
- The python3 block should read directories and files directly (using os, pathlib) rather than receiving data from bash pipes -- this is more robust for 160+ agents

**Error handling:**
- Missing directories (~/.claude/agents/, ~/.claude/skills/, ~/.claude/hooks/): produce empty arrays, no error exit
- Missing ~/.claude/settings.json: produce empty MCP array (from enabledPlugins source)
- Missing ~/.claude/installed_plugins.json: skip that source, no error (only use enabledPlugins from settings.json)
- Malformed YAML frontmatter in an agent file: skip that entry, continue scanning (log warning to stderr)
- Script must always exit 0 (consumed by wizard pipeline)
- Error output to stderr, structured data (compact summary) to stdout only

**Also update `.gitignore`:**
- Add `toolkit-registry.json` entry under a new comment section `# Toolkit discovery (machine-specific)`
- Place after the existing `# Local configuration` section

Make the script executable: `chmod +x skills/toolkit-discovery.sh`
