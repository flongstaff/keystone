Create `skills/wizard-backing-agent.md` following the conditional operation block pattern from bmad-gsd-orchestrator.md. The file has three sections: YAML frontmatter, route dispatch, and two route blocks.

**YAML frontmatter:**
```yaml
name: wizard-backing-agent
description: >
  Backing agent for the wizard. Handles Route A (resume GSD/BMAD work)
  and Route B (bridge from completed BMAD to GSD with traceability assertions).
model: sonnet

  - Read
  - Write
  - Edit

  - AskUserQuestion

```

**Route Dispatch section:**
Read `.claude/wizard-state.json`. Determine route:
- If `gsd.present == true` AND scenario is `full-stack` or `gsd-only`: follow Route A (Resume GSD)
- If scenario is `full-stack` and `bmad.present == true` and all BMAD docs complete (prd + architecture + stories_approved == stories_total) and user requested bridge: follow Route B
- If scenario is `bmad-only`: follow Route B (Bridge to GSD)
- If scenario is `bmad-only` but BMAD is NOT complete (prd false, arch false, or stories_approved < stories_total): follow Route A variant for BMAD resume
- Default fallback: display diagnostic and suggest running `/wizard` again

**Route A -- Resume GSD/BMAD Work:**

1. Read `.claude/wizard-state.json` (already done in dispatch)
2. Determine if this is a GSD resume or BMAD resume:

For GSD resume (gsd.present == true):
- Read `.planning/STATE.md` to get `stopped_at` and `last_activity` from YAML frontmatter
- Read `.planning/ROADMAP.md` and grep `### Phase {gsd.current_phase}` to get the phase name
- Display orientation context:
  ```
  Picking up where you left off:

    Phase {N} of {total}: {phase name from ROADMAP.md}
    Last activity: {stopped_at from STATE.md}
    Status: {phase_status}

    Next: {next_command}
  ```
- Then auto-invoke the next_command: try `Skill('{command}')` first, fallback to reading the command file at `.claude/commands/{command-name}.md`
- CRITICAL: Always read `next_command` directly from wizard-state.json. Never re-derive the next GSD command.

For BMAD resume (bmad.present == true, gsd.present == false, BMAD NOT complete):
- Read wizard-state.json bmad.* fields
- Display what's complete and what's missing:
  ```
  BMAD Planning Status:
    PRD: {done/missing}
    Architecture: {done/missing}
    Stories: {approved}/{total} approved

  Next step:
  ```
- Suggest appropriate BMAD command based on what's missing:
  - `bmad.prd == false` -> suggest `/analyst` or `/pm`
  - `bmad.architecture == false` -> suggest `/architect`
  - `bmad.stories_approved < bmad.stories_total` -> suggest `/po` or `/sm`
- Offer to invoke the suggested command

**Route B -- Bridge to GSD + Traceability Assertion:**

Step 1 - BMAD Eligibility Gate:
- Read wizard-state.json bmad.* fields
- Check: `bmad.prd == true` AND `bmad.architecture == true` AND `bmad.stories_total > 0` AND `bmad.stories_approved == bmad.stories_total`
- If NOT eligible: display what's missing, suggest the appropriate BMAD command, STOP (do not bridge incomplete planning)
- If `bmad.stories_total == 0`: use AskUserQuestion to ask "No story files found. Proceed with bridge without story-level acceptance criteria? (The PRD and architecture will still be used.)" -- handle the zero-story edge case explicitly per research pitfall 3

Step 2 - Prompt before bridging:
- Display eligibility confirmation: "BMAD planning is complete. Ready to bridge to GSD execution."
- Use AskUserQuestion: "Proceed with bridge? This will create the .planning/ execution structure from your BMAD docs."
- If user declines, stop gracefully

Step 3 - Delegate to bmad-gsd-orchestrator via Task():
- Use the Task tool to spawn bmad-gsd-orchestrator in a fresh context (satisfies ORCH-02):
  ```
  Task description: "Bridge BMAD planning docs to GSD execution structure"
  Task prompt: "Read agents/bridge/bmad-gsd-orchestrator.md and follow Operation A to initialise GSD from BMAD docs."
  ```
- Do NOT reimplement Operation A logic (satisfies ORCH-01). The backing agent is a coordinator, not a reimplementor.

Step 4 - Verify bridge success:
- After Task() returns, verify by checking for expected file artifacts:
  ```bash
  test -f ".planning/config.json" && test -f ".planning/CONTEXT.md" && echo "BRIDGE_COMPLETE=true" || echo "BRIDGE_COMPLETE=false"
  ```
- If bridge failed, display error and suggest re-running

Step 5 - Traceability Assertion:
- Find all story files:
  ```bash
  find docs/stories _bmad-output/stories -maxdepth 1 -name "story-*.md" 2>/dev/null
  ```
- For each story file, extract acceptance criteria using awk (handles both H2 and H3 headings per research pitfall 2):
  ```bash
  awk '/^#{1,3} [Aa]cceptance [Cc]riteria/{flag=1; next} /^#{1,3} /{flag=0} flag && /^- /{print}' "$STORY"
  ```
- Extract only top-level AC bullets (lines matching `^- `). Sub-bullets are details, not separate criteria (per research pitfall 1).
- For each extracted AC, check presence in .planning/ files (cast wide -- search all .planning/ including context/ and phases/):
  ```bash
  AC_TEXT=$(echo "$AC" | sed 's/^- //')
  FOUND=$(grep -rl "$AC_TEXT" .planning/ 2>/dev/null | wc -l | tr -d ' ')
  ```
- Collect ALL missing ACs first before presenting to user (do not block after first gap -- per research anti-pattern)
- Note: string-match anywhere in a context file is an acceptable approximation for Phase 4 (per research pitfall 5)

Step 6 - Interactive Gap Resolution:
- If any ACs are missing, present the FULL list of gaps first
- Then for each missing AC, use AskUserQuestion:
  ```
  Criterion not covered: '{AC text}' (from {story name})

  Choose:
  1. Map to Phase {N} -- I'll add it to that phase's context file
  2. Explicitly defer -- this criterion is out of scope for this milestone
  3. I'll handle it manually -- mark as acknowledged
  ```
- Option 1: Edit the specified `.planning/context/phase-N-context.md` to add the criterion under Acceptance Criteria
- Option 2: Append to `.planning/DEFERRED-CRITERIA.md` tracking file with the criterion text, story source, and "deferred" status
- Option 3: Append to `.planning/DEFERRED-CRITERIA.md` with "acknowledged" status
- CRITICAL: Never silently skip a criterion. Every AC must be found, mapped, or explicitly deferred (per locked decision).

Step 7 - Bridge Complete:
- Display confirmation:
  ```
  Bridge Complete.
  {N} phases created from BMAD planning docs.
  {M} acceptance criteria verified ({X} found, {Y} mapped, {Z} deferred).

  Next: /gsd:discuss-phase 1
  ```
- Stop and show result. Do NOT auto-invoke /gsd:discuss-phase 1 (maintains auto_advance: false discipline per research recommendation).

**Rules section at the end of the file:**
- Never write to wizard-state.json -- wizard-detect.sh owns writes
- Never reimplement bmad-gsd-orchestrator logic -- delegate via Task()
- Always read next_command from wizard-state.json -- never re-derive it
- Traceability assertion extracts top-level AC bullets only (^- prefix)
- A gap is not a build failure -- it is a question for the user
